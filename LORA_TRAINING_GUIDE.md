# LoRA 微调完整指南：从数据到模型的版本迭代

## 一、训练流程总览

```
PubMed检索 → LLM标注PICOS → MLX格式转换 → LoRA训练 → 模型融合 → 效果验证
    ↓            ↓               ↓              ↓           ↓          ↓
 30个检索词    DeepSeek      {"messages":   mlx_lm lora   fuse命令   subprocess
 每词20篇      temperature   [{system..    -c config     合并权重    对比测试
 收集543篇     0.1批量标注    user..       生成adapters  →完整模型
                             assistant}]
```

### Step 1: 数据收集（scripts/expand_dataset.py）
- PubMed E-utilities API 检索 30 个不同临床领域的 RCT/队列研究
- 去重后用 DeepSeek API 批量标注 PICOS（temperature=0.1 保证一致性）
- 每 50 条自动保存，防止中断丢失

### Step 2: 格式转换
- MLX 要求的格式：`{"messages": [{"role": "system", ...}, {"role": "user", ...}, {"role": "assistant", ...}]}`
- 每个 sample 包含 system prompt + 用户摘要 + JSON 格式的 PICOS 标签
- 85/15 随机分割为 train/valid

### Step 3: LoRA 配置
- 配置文件：`lora_config.yaml`
- 核心超参：rank, scale, dropout, lr, grad_accum, iters, seq_len
- 使用 `-c` 标志加载 YAML 配置

### Step 4: 训练
- `python -m mlx_lm lora -c lora_config.yaml`
- MLX 自动利用 M4 GPU + Neural Engine
- 每 10 步报告 train loss，每 50 步报告 val loss
- 每 100 步保存 checkpoint

### Step 5: 融合与评估
- `python -m mlx_lm fuse` 将 LoRA 权重合并到基座模型
- 用 subprocess 调用 mlx_lm.generate 对比微调前后效果

## 二、三版迭代对比

| 参数 | V1 (失败) | V2 (小数据优化) | V3 (大数据) |
|------|-----------|-----------------|-------------|
| **训练数据** | 40 条 | 40 条 | **473 条** |
| **验证数据** | 8 条 | 8 条 | **84 条** |
| **LoRA rank** | 32 (默认) | **8** | **8** |
| **scale** | 1.0 (alpha=32) | **2.0** | **2.0** |
| **dropout** | 0.0 (默认) | **0.1** | **0.1** |
| **learning rate** | 1e-5 | **5e-6** | **5e-6** |
| **grad accumulation** | 1 | **4** | **2** |
| **max_seq_length** | 2048 | 2048 | **1536** |
| **iters** | 500 | **200** | **300** |
| **可训参数** | 5.28M (0.34%) | 5.28M (0.34%) | 5.28M (0.34%) |
| **峰值内存** | 10.8GB | 10.8GB | 10.6GB |
| **Train loss 最终** | 0.026 (逼近0) | 1.378 | **1.255** |
| **Val loss 最终** | 3.532 (飙升) | 1.158 | **1.397** |
| **Train/Val 差距** | 3.5+ (背离) | -0.22 (反转) | **0.14 (收敛)** |
| **过拟合** | 严重 | 无 | **无** |
| **泛化评估可信度** | — | 低 (8条验证) | **高 (84条验证)** |

### V1 → V2：诊断并解决过拟合

**V1 的问题**：40 条数据，rank=32，dropout=0，lr=1e-5，500 步。训练损失降到 0.026（几乎记住所有答案），验证损失飙升到 3.532（比不训练还差）。

**根因分析**：
- 数据太少（40 条）无法约束 5.28M 参数
- 没有正则化（dropout=0）
- 学习率偏高，在小数据集上快速过拟合
- 训练步数过多（500 步，但最佳在 200 步）

**V2 的修复**：
- rank 32→8：减少可训参数的自由度（虽然实际参数数没变，因为新版默认应用到更多层）
- dropout 0→0.1：训练时随机丢弃 10% LoRA 神经元，强制模型不依赖特定权重
- lr 1e-5→5e-6：步长减半，防止在损失曲面跳过头
- grad_accum 1→4：积累 4 步梯度再更新，有效 batch=4，梯度方向更平滑
- iters 500→200：提前停止，不在过拟合区域训练

**V2 效果**：验证损失从飙升变为持续下降（1.87→1.16），train loss 保持在 1.4，不再逼近 0。但只有 8 条验证，结果不够可信。

### V2 → V3：扩充数据到 473 条

**V3 的改动**：
- 数据 40→473（11.8 倍），验证 8→84（10.5 倍）
- 30 个多样化检索词覆盖心血管、糖尿病、精神科、呼吸、肿瘤等
- grad_accum 4→2（数据多了，梯度本身就更稳定）
- max_seq_length 2048→1536（解决大样本导致的 OOM）
- iters 200→300（数据多可以多训）

**V3 效果**：
- Train loss 1.255，Val loss 1.397，差距仅 0.14
- 曲线平稳收敛，无过拟合信号
- 峰值内存 10.6GB，16GB M4 完全够用

## 三、核心技术细节

### 为什么用 LoRA 而不是 Full Fine-tuning？

LoRA 的思想是冻结预训练权重 W，只训练低秩增量 ΔW = B × A（A∈R^(r×d), B∈R^(d×r)）。

```
原始: h = Wx
LoRA: h = Wx + scale × BAx
```

- **参数效率**：Qwen-1.5B 有 15.4 亿参数，LoRA 只训练 528 万（0.34%）
- **内存友好**：10.6GB 峰值，16GB M4 就够，不需要 A100
- **防止灾难性遗忘**：99.66% 权重不变，模型保留预训练知识
- **快速迭代**：每次训练 5-10 分钟，可以快速实验

### rank 和 scale 的关系

在 LoRA 论文中：`ΔW = (α/r) × B × A`

- rank (r) 控制低秩矩阵的维度，rank 越大可训参数越多
- scale (α/r) 控制 LoRA 更新的幅度，新版 mlx-lm 直接暴露 scale
- scale=2.0 意味着 LoRA 的贡献被放大 2 倍
- 原理：r 小了 → 可训参数少 → 表达力弱 → 需要更大的 scale 来补偿

### 为什么 dropout 能防过拟合？

训练时随机丢弃 10% 的 LoRA 神经元 → 模型不能依赖特定参数的组合 → 被迫学习多种等效特征 → 泛化能力更强。在小数据集上（40 条），dropout 是最有效的正则化手段。

### 为什么 grad_accumulation 有效？

batch_size=1 时每一步梯度噪声很大。grad_accum=4 让模型走 4 步积累梯度后取平均再更新，等效于用 4 条数据同时训练。区别是梯度累积省显存（同一时刻只需 1 条数据的计算图）。

V2 用 grad_accum=4（数据少，需要更大的有效 batch 来平滑梯度），V3 降到 2（数据多，单条梯度本身就更多样）。

### 为什么 V3 的 seq_len 需要降到 1536？

V1 的 OOM 原因是：某些摘要超长（2405 tokens）+ grad_accum=4 = 需要同时保持多条长序列的计算图 → 超过 16GB 内存。设置 max_seq_length=1536 将序列截断，超长部分丢失（但 PICOS 信息通常在摘要的前 1000 tokens 内）。

## 四、面试 Q&A

### Q1: 为什么选择 LoRA 微调而不是全量微调？

> 三个原因。首先是硬件限制：Qwen2.5-1.5B 全量微调需要约 28GB 显存（Adam 优化器需要额外存储一阶和二阶动量），我的 M4 MacBook 只有 16GB，跑不动。其次是效率：LoRA 只训练 0.34% 参数，每轮实验 5-10 分钟，如果全量微调每次要 1 小时以上，无法快速迭代。最后是数据量匹配：即使大数据集也只有 473 条，全量微调 15 亿参数必然过拟合，LoRA 的参数少得多，和 473 条数据的规模是匹配的。

### Q2: 你做 LoRA 的时候遇到了过拟合，怎么诊断和解决的？

> 诊断方法很直接：看 train loss 和 val loss 的走势。V1 的 train loss 从 1.5 降到了 0.026（快接近 0 了），但 val loss 从 1.87 飙升到 3.53。两条曲线一个向下一个向上，经典过拟合——模型在训练集上完美但验证集上完全不行。

> 解决分两个阶段。第一阶段在不扩充数据的前提下做正则化优化：rank 从 32 降到 8、加 dropout 0.1、lr 从 1e-5 降到 5e-6、加 grad_accum 让梯度更平滑。这些改动把 val loss 从 3.53 降到了 1.16，在不增加数据成本的前提下解决了问题。第二阶段扩充数据的逻辑是：正则化只能防止过拟合，不能提供新信号。要提升泛化能力，最终还是需要更多数据。

### Q3: 为什么 rank=8 就够了，而不是更高？

> LoRA 的 rank 比大家想象的低。原论文实验表明 rank=4 或 8 在很多任务上就够了，因为预训练模型本身已经学到了丰富的表征，微调只需要在一个低维子空间里做方向性调整。rank 的决定因素是任务复杂度和数据量，不是模型大小。PICOS 提取是一个结构化 JSON 输出任务，比开放式文本生成简单很多，rank=8 的 528 万参数已经够用。rank=16 或 32 反而在小数据下更容易过拟合，我在 V1 实验里验证了这一点。

### Q4: 你用的训练数据只有 473 条，够吗？为什么不加更多？

> 473 条对于 PICOS 这个特定任务来说足够了。LoRA 只需要少量数据——原论文用几百条就取得了好效果。PICOS 提取是一个格式固定的结构化任务，输出空间很小（5 个固定字段），几百条高质量的多样样本足够让模型学到模式。如果再扩大数据，边际收益会递减。关键不是数量而是覆盖度：我的 30 个检索词覆盖了心血管、糖尿病、精神科、呼吸、消化、肿瘤、感染等多个领域，确保模型见过来自不同领域的 PICOS 格式。

### Q5: 你在 M4 MacBook 上训练遇到内存不足怎么解决的？

> V1 跑完没问题但 V3 数据多了就 OOM，根因是某些摘要超过 2000 token，加上 grad_accum=4 要同时保持 4 条长序列的计算图，峰值超过 16GB。解决：第一，把 max_seq_length 从 2048 降到 1536，超长部分截断（PICOS 信息通常在摘要前半部分）；第二，把 grad_accum 从 4 降到 2，减少同一时刻的计算图大小。两个改动把内存从 OOM 压到 10.6GB。如果再不够还有 grad_checkpoint 可以用——用计算换内存，因为之前不需要所以没开。

### Q6: mlx-lm 的版本从旧到新有什么变化，你怎么处理的？

> 新版 mlx-lm v0.31.3 重构了 LoRA 参数管理。旧版用独立的 `--lora-rank`、`--lora-alpha` 命令行标志，新版统一为 `lora_parameters` 字典放在 YAML 配置里，且用 scale 替代了 alpha。API 变好用了——scale 直接控制更新幅度，不需要心算 alpha/rank。我通过读 mlx_lm 源码里 CONFIG_DEFAULTS 确认了新格式，然后更新了 yaml。

### Q7: 你怎么判断模型训练好了而不是欠拟合或过拟合？

> 三个指标同时看：1) train loss 是否降到合理范围（不能逼近 0，1.2 左右是健康的）；2) val loss 是否持续下降（不能回升）；3) train/val 差距是否收敛（不能扩大）。V3 的 train=1.255, val=1.397, gap=0.14，三者都满足，说明训练是健康的。如果 train loss 还在高位（>2.0），说明欠拟合；如果 val loss 回升，说明过拟合开始。

### Q8: 整个流程中最大的挑战是什么？

> 不是技术问题，而是数据质量控制和实验设计的迭代思维。这个项目本质上是 Small-to-Large 架构中 Small model 的定制化，每个决策都需要权衡——数据量 vs 标注成本、正则化 vs 训练速度、rank vs 过拟合风险。最关键的判断是 V1 过拟合后，我没有盲目加数据，而是先优化正则化验证方向正确，再从容扩充数据。这种"先验证假设，再投资成本"的实验思维，在实际工作中能避免很多返工。

### Q9: 你的 Small-to-Large 架构为什么要自己微调 small model，不能直接用 GPT/DeepSeek 吗？

> 两个原因。成本和延迟：系统每篇论文都需要提取 PICOS，一次 Meta 分析可能筛选上百篇。用 DeepSeek 每次 $0.0002，每篇 1 秒，一百篇 $0.02 还行。但如果是上千篇或需要频繁重跑，成本就上来了。本地模型一次加载后无限调用，单次推理 <0.5 秒、零费用。更重要的是架构意义：Small-to-Large 的核心是"把简单且高频的任务交给小模型，复杂且低频的 GRADE 推理交给大模型"。这是资源优化，不是功能妥协。

### Q10: 如果让你把这个模型部署到生产环境，还需要做什么？

> 三件事。第一，量化——MLX 支持 4-bit/8-bit 量化，可以把 2.9GB 模型压到 1GB 以下，推理更快。第二，换用 MLX Python API 替代 subprocess 调用——当前 extract_agent 每次 subprocess 启动都重新加载模型，应该改为加载一次后循环调用。第三，增加监控和 fallback——当本地模型提取失败或 confidence 低时，自动 fallback 到 DeepSeek，保证系统鲁棒性。这三步做完就可以部署了。
